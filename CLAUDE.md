## Project Overview

One Page is a Flutter diary app with a unique "one page only" concept. It uses a monorepo architecture with Melos for package management and is organized into feature-based packages.

## Development Setup

```bash
make            # Initial setup (installs FVM, Melos, and required tools)
make worktree   # Worktree setup
```

Use FVM for Flutter version management:
- **Always prefix direct `dart` / `flutter` invocations with `fvm`** (e.g. `fvm dart analyze`, `fvm flutter test`). The `dart`/`flutter` on PATH may point to a different SDK than the project's pinned version (`.fvmrc`).
- **Run melos via the workspace dev dependency: `fvm dart run melos <command>`** (e.g. `fvm dart run melos run gen`). The globally activated `melos` binary runs on the PATH SDK and fails when that SDK does not satisfy the workspace constraint; the `sdkPath` in the melos config only pins the SDK for melos's child processes, not for melos itself. (CI is unaffected — it sets `MELOS_SDK_PATH` explicitly.)

## Architecture

### Package Structure
- **app/**: Main application package with UI, routing, and app-level providers
- **core/**: Shared utilities and foundational packages
  - **db_client/**: Drift-based SQLite database client
  - **i18n/**: Slang-based internationalization (English/Japanese)
  - **provider_utils/**: Riverpod utilities and common providers
  - **theme/**: App theming and design system
  - **utils/**: General utility functions
  - **widgets/**: Shared UI components (domain-independent only)
- **features/**: Domain-specific functionality
  - **backup/**: Data backup and restore
  - **diary/**: Core diary functionality
  - **scroll_calendar/**: Calendar widget
  - **update_requester/**: App update management

Detailed package placement, dependency, and DI rules: `.claude/rules/architecture.md`.

### State Management
- Uses Riverpod with code generation (`@riverpod` annotation), generated by `riverpod_generator`.
- Widget selection and `ref.watch` / `ref.read` usage: see `.claude/rules/flutter-widgets.md`.

### Database
- SQLite with Drift ORM, schema versioning with migrations, client provided through dependency injection.

## Internationalization
- Slang framework for type-safe translations; English (`app_en.yaml`) and Japanese (`app_ja.yaml`).
- **NEVER** use hardcoded Japanese text in Dart files (enforced by `avoid_hardcoded_japanese` custom lint).
- **ALWAYS** add strings to both `packages/core/i18n/lib/src/i18n/app_ja.yaml` and `app_en.yaml`.
- Access via `context.t.section.key`. Run `fvm dart run melos run slang` after updating YAML files.

## Code Generation Dependencies
Run `fvm dart run melos run gen` after changes to:
- Riverpod providers with `@riverpod`
- Drift database tables
- i18n translation files
- Freezed data classes

## Error Handling & Validation
- Check `context.mounted` before async operations.
- Always validate date ranges and scroll positions.
- **ScrollCalendar limits**: `VerticalScrollCalendar` has a limited date range. Check if the target date is within the loaded range before `scrollToDate()`, use `loadMoreOlder()` to expand it, and wrap scroll operations in try-catch.

## Environment Configuration
- Environment variables in `packages/app/dart_defines/`.
- Firebase configuration files for dev/prod environments.
- Use `--dart-define-from-file` for builds.
